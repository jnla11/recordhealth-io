Why this exists
The mission
A true patient-advocate guiding their health journey using evidence based medicine and removing layers of abstraction and errors that lead to negative health outcomes. Available in all languages.
— Jason Nolte

The five layers being removed

  • 1
    Document layer
    Records scattered across portals and folders → assembled from every document, every provider.
  • 2
    Comprehension layer
    Numbers without context → trends and patterns surfaced with clinical meaning.
  • 3
    Language layer
    English-only healthcare → any language in, any language out.
  • 4
    Coordination layer
    Each provider has a fragment → one longitudinal patient model across all of them.
  • 5
    Evidence layer
    Clinical guidelines never reach the patient → evidence-based questions the patient can act on.
The loop
How the recursion works
Two completely separate systems. System A trains offline. System B runs on device. System A output feeds System B. System B never feeds System A automatically.
SYSTEM A — Offline training (AWS, no live traffic) Expert annotation → training run → benchmark gate → promote artifact ↓ SYSTEM B — On-device production (live users) Receives pre-validated rules + model → runs extraction → user sees result User flags something → anomaly review queue → L0 decides NOT → automatic training → NOT → PatternLibrary

Three device layers

Layer 1 — Rules
PatternLibrary deltas. Cross-user. Frequent silent sync. Explicit and inspectable.
Layer 2 — Model
Fine-tuned CoreML. Cross-user. Periodic staged rollout. Generalizes beyond rules.
Layer 3 — Personal
MLUpdateTask on device. This user only. Continuous. Never transmitted.

The recursion cycle

Annotate → add expert-labeled examples to corpus Train → AWS fine-tuning run produces candidate model Benchmark→ score candidate against held-out test silo Gate → passes threshold → route to delivery channel Ship → rules or model lands on device Distill → model learns → extract rules it discovered Repeat → better extraction → fewer errors → higher-signal annotations

The rules
These never change. If a sprint prompt conflicts with any of these, the prompt is wrong.
  • ✓
    PHI never leaves the device in raw form
    Tokenization happens before any transmission. Always.
  • ✓
    FactStore is append-only
    No updates. No deletes. Full provenance chain preserved forever.
  • ✓
    Atomic writes only
    No try?. Failures surface to the caller, never silently swallowed.
  • ✓
    Audit before implement
    Every sprint reads existing code before writing a line of new code.
  • ✓
    MIMIC source content stays local
    Never transmitted to any external API. Tokenized extraction only crosses the wire.
  • ✓
    Test silo is locked
    Never annotated, never trained on. Used only to score benchmark runs.
  • ✓
    Bedrock is inference only
    Never training. Stateless. Nothing persists after a call.
  • ✓
    Every Bedrock suggestion requires your approval
    Nothing enters the corpus automatically.
  • ✓
    pattern_library is only written by the consensus CRON
    No component writes to it directly. All promotions go through the gate.
  • ✓
    User corrections are not cross-user training signal
    They feed anomaly detection only. Expert annotation is the sole source of cross-user training data.
    ✕ Never write user corrections to pattern_atoms directly
    ✕ Never pipe user corrections to candidate_rules
  • ✓
    Correction form UI is admin-only
    Class A–E correction interface lives in SEED Master Control only. End users see a simple flag button.
  • ✓
    Stage transitions are manual
    No automatic phase advancement. Requires your explicit sign-off.
  • ✓
    Never touch animateIn() in DashboardView.swift
    Never touch CareGraphBuilder.swift line 390 — P6b deferred.

Build step 3
Annotate
The annotation session is the core of the operation. Bedrock pre-fills the form. You confirm, modify, or reject. Your judgment is the gate.
  • 1
    Open SEED Master Control → Annotation queue
    Queue loads highest-value documents first (active learning ordering). Each document is pre-annotated by Bedrock before you open it — forms are pre-filled.
  • 2
    Review each extraction against the source document
    Bounding boxes on the rendered PDF: amber = unreviewed, red = flagged wrong, green = confirmed, blue dashed = missing field. Click a box to load it into the correction form.
  • 3
    For each extraction: accept, modify, or reject Bedrock's suggestion
    Accept → hit Enter (8 seconds). Modify → change the class or rationale, then save. Reject → select correct class yourself. Use keyboard shortcuts: 1–5 for correction class, Enter to save and advance.
  • 4
    Flag negative space — fields present in document but not extracted
    Press N → add missing field form → select field type, canonical ID, clinical significance. This is Class E — the most clinically important correction type and the hardest to catch.
  • 5
    Complete session → generates PatternAtoms
    SEED_CORPUS_STARTER_KIT.md §7.2
    POST /v1/seed/generate-atoms runs PHI strip verification and writes pattern_atoms from your completed annotations. PHI strip verified before any atom enters the pipeline.

Build step 4
Train
All training is offline. Spin up an instance, run the job, terminate. The training environment never touches live user traffic.
  • 1
    Export annotation corpus to JSONL
    FUNCTION_DISTINCTION.md §5.1
    export_corpus.py reads expert_annotations from Neon (not CorrectionStore), outputs training/validation JSONL files. Each record: input = document region context, output = structured extraction JSON.
    ✕ Do not export from CorrectionStore — user corrections are not training data
  • 2
    Launch AWS training instance
    EXPERT_ANNOTATION_AND_MODEL_TRAINING.md §5.2
    Stage 2: g4dn.2xlarge spot (~$0.23/hr). 2–4 hour run = ~$1. Stage 3: g4dn.12xlarge. Stage 4: p3.8xlarge. Use spot instances for training — terminate when done, zero ongoing cost.
  • 3
    Run LoRA fine-tuning script
    Hugging Face Trainer + PEFT library. Base model: Phi-3 Mini at Stage 2. LoRA config: r=16, lora_alpha=32, target_modules q_proj/v_proj. 3–5 epochs. Monitor loss curve — stop before overfitting.
  • 4
    Evaluate candidate against validation silo
    SEED_CORPUS_STARTER_KIT.md §8.2
    eval.py runs candidate model against validation corpus. Computes F1 per field type. Compare to current production model. If positive delta → proceed to gate. If flat or negative → investigate before proceeding.
  • 5
    Run production gate against test silo
    SEED_CORPUS_AND_TAXONOMY.md §4.9
    gate.py runs candidate against held-out test silo. Must clear all field-type thresholds. No regression vs prior approved version. Gate result logged to benchmark_runs. If approved → write training_artifacts row.
  • 6
    Run distillation analysis
    FUNCTION_DISTINCTION.md §5.2
    Compare new model against rule engine on validation corpus. Where model wins → can this pattern be expressed as an explicit rule? If yes → generate candidate rule → enters PatternLibrary pipeline as Route A. Faster to device than the model itself.
  • 10
    Step 10 — Sampled Bedrock recursion + context expectation engine
    SEED_CORPUS_RECURSIVE_TRAINING_STARTER_KIT.md §13 — build after 200+ annotated docs
    Two self-improving mechanisms. (1) Sampled Bedrock recursion: statistically samples uncertain extractions (confidence < 0.75), runs Bedrock review pass, batch-approves high-confidence corrections into corpus at 0.7x weight. Cap 50 docs/run — not full recursion. Re-samples automatically when Bedrock model version upgrades. (2) Context expectation engine: mines co-occurrence patterns across corpus, learns what each document type should contain, tightens ingest pass proactively. Conservative activation — expectations only set when sample size is sufficient.
    ⚠ Build after benchmark_runs has 10+ stable runs and corpus has 200+ annotated documents
    ✕ Full recursion on every document — cost and signal quality both fail
    ✕ Context expectations from thin data — activation thresholds are hard limits

Build step 5
Promote
The gate decides what ships and where. Every artifact is routed before distribution begins. You approve — nothing promotes automatically.
  • 1
    Open SEED Master Control → PatternLibrary screen
    Two queues: seed corpus candidates (from expert annotations) and Bedrock suggestions (terminology, rules, classification disagreements). Both require your approval before anything promotes.
  • 2
    Review each candidate — approve or reject
    SEED_CORPUS_STARTER_KIT.md §9.2
    POST /v1/admin/seed/approve → candidate promotes at next 2AM CRON. POST /v1/admin/seed/reject → logged, no promotion. Approved seed candidates bypass the validation window — manually verified by you.
  • 3
    Confirm artifact route before distribution
    FUNCTION_DISTINCTION.md §4
    PatternLibrary rule delta → Route A (silent device sync). CoreML model update → Route B (staged rollout). Private instance update → Route C (blue/green). Schema change → Route D (App Store). Routing determined automatically — you confirm it makes sense.
  • 4
    For Route B model updates — initiate staged rollout
    FUNCTION_DISTINCTION.md §4.2
    10% of users day 1 → monitor anomaly_flag rate delta vs control group. 50% day 3 if positive. 100% day 7 if positive. Halt and rollback if flag rate increases. Prior model retained on device for 14 days.

Build step 6
Ship to device
Two delivery mechanisms for device intelligence. Both happen silently. Users get better extraction without knowing anything changed.
  • 1
    PatternLibrary delta sync (Route A)
    ADAPTIVE_DOCUMENT_INTELLIGENCE.md §10.3
    iOS app checks version on app open. If behind: download delta, validate patch signature, apply atomically, trigger retroactive re-scoring on prior documents under the old version. Silent. No user action. Happens every app open if there's a new version.
  • 2
    CoreML model download (Route B)
    Host compiled .mlmodelc on CloudFront. App checks for new model version on open (same pattern as PatternLibrary sync). Background download. Hot-swap via MLModel(contentsOf:). No App Store update for architecture-unchanged model updates.
  • 3
    Monitor real-world performance during Route B staged rollout
    Watch anomaly_flag rate in the SEED Master Control dashboard. Flag rate should stay flat or decrease after a model update. If it increases — the new model is worse in the real world regardless of benchmark scores. Halt and rollback immediately.
    ⚠ Real-world flag rate is ground truth. Benchmark approximates it.
  • 4
    Retroactive re-scoring triggers automatically
    ADAPTIVE_DOCUMENT_INTELLIGENCE.md §10.4
    When a new library version applies, FactStore is queried for extractions under prior versions. Affected fields re-run against new rules. High-confidence improvements auto-accept. Lower-confidence improvements surface for user review. Users' old documents get smarter.
Operations
Weekly rhythm
Monday is the operational day. Everything automated runs overnight. You review and act in the morning.
  • 1
    6:00 AM UTC — Automated benchmark runs
    30 documents, training split. Benchmark scores written to benchmark_runs. Bedrock analysis chain fires automatically: failure classification, terminology gaps, rule suggestions. All queued in /v1/admin/seed/analysis-review.
  • 2
    Morning — Open SEED Master Control
    Dashboard shows: accuracy delta from prior week, pending review counts, system maturity phase, silent miss rate. Review Bedrock classification disagreements → approve/reject terminology suggestions → approve/reject rule suggestions.
  • 3
    Session — Annotation queue
    Active learning serves highest-value documents first. Target: 80–90 documents/hour with Bedrock pre-annotation. Focus on the failure class the benchmark flagged as worst this week. Each annotation session directly funds the next training run.
  • 4
    Review PatternLibrary candidates
    GET /v1/admin/seed/candidates — any candidates that cleared the threshold appear here. Approve → promotes at next 2AM CRON. Keep approval sessions short — if something looks wrong, reject and investigate.
  • 5
    Review user anomaly flags
    GET /v1/seed/anomaly-flags/pending. Classify each: misunderstanding, known class, novel failure, or data quality. Only novel failures create expert_annotations rows. Everything else is closed. Most weeks this is under 10 flags total.

Operations
Stage gates
Phase transitions are benchmark-driven, not user-count-driven. Every transition is manual — you sign off before the system advances.
  • 0
    Phase 0 → Phase 1
    FUNCTION_DISTINCTION.md §2.2
    Expert annotation corpus ≥ 500 Tier A documents AND benchmark overall F1 > 0.61 on test silo. System is ready for limited use.
    ⚠ Do not open to users before F1 > 0.61
  • 1
    Phase 1 → Phase 2 (acceptable UX)
    FUNCTION_DISTINCTION.md §2.2, §7.1
    Benchmark overall F1 > 0.78 on test silo AND Tier 1 auto-accept rate > 82% on validation silo. System works well enough that users don't need to correct it to trust it.
  • 2
    Stage 2 → Stage 3 (document-level model)
    EXPERT_ANNOTATION_AND_MODEL_TRAINING.md §4.3
    3,000 annotated documents, 1,500 Tier A, fine-tuned model outperforms rule engine on test silo, L4 informaticist has reviewed annotation schema.
  • 3
    Stage 3 → Stage 4a (clinical reasoning)
    7,000+ annotated documents, MIMIC-IV access active, private AWS instance operational, HIPAA BAA with AWS confirmed for EC2, institutional partnership or funding secured.
    ⚠ Engage FDA regulatory counsel before Stage 4 — not after
  • 4
    Stage 4a → Stage 4b (consented corpus)
    Tier 3 consent framework legally reviewed, IRB determination completed, research DUA executed. Consented corpus reaches 10,000+ documents. MIMIC becomes validation baseline, not training source.
    ✕ Do not activate Tier 3 without regulatory counsel sign-off
