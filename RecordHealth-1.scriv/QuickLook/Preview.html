      • Cloud Vault

        Cloud vault is an encrypted sync + compute adjunct (backup, multi-device, heavy lifting).

        • Cloud storage: sync without giving the cloud your plaintext (if possible)

          Best-practice target: client-side encryption (zero-knowledge style)

           • Device generates a Data Encryption Key (DEK) per file (or per chunk).

           • Encrypt file locally → upload encrypted blob to cloud object storage.

           • Cloud never sees plaintext.

          Key management options

           • User-derived key (passphrase) → you can’t recover accounts easily.

           • Escrowed keys (recoverable) → more supportable, but higher trust burden.

        • LLM cloud: “minimize exposure” design (RAG + strict controls)

          You basically have three patterns:

          Pattern 1: “Cloud LLM sees plaintext” (simplest, most risk)

           • You send chunks of PHI to the LLM API.

           • Requires extremely strong vendor terms and technical controls.

           • In HIPAA mode, you’ll need a BAA (and likely a dedicated/isolated setup).

          Pattern 2: “De-identify/redact → cloud LLM” (middle ground)

           • On-device (or in your secure compute), redact identifiers:

           ◦ Names, addresses, MRN, phone, DOB...

      • Synthesis

        Synthesis (summaries/timelines/insights) is treated as derived data that can be regenerated.

        • Retrieval-Augmented Generation (RAG): keep prompts small and auditable

          Instead of sending the entire record set:

          Pipeline

           1 User asks a question

           2 Local (or secure server) retrieval pulls top relevant chunks

           3 Only those chunks (minimal necessary) are sent to the LLM

           4 Response includes citations (links back to exact source docs/pages)

    • 2. Data Layers

      • Raw Records

        Raw records (immutable)

         • PDFs, images, CCDA, FHIR bundles, lab results, etc.

         • Stored encrypted locally and encrypted in cloud object storage.

      • Normalized “health graph”

    • 3. Security

      • Encryption & key model


        Recommended: client-side encryption + split-key recovery

         • Each document/chunk encrypted with a DEK (data key).

         • DEKs are wrapped by a KEK (key-encryption key).

         • KEK is split:

         ◦ Part protected on-device (Secure Enclave/Keystore)

         ◦ Part stored in cloud KMS/HSM (tenant-isolated)

         • Decryption requires both parts + authenticated user session.

        Why this hybrid key approach is good:

         • Cloud can store and sync safely (it holds encrypted blobs).

      • Security Gameplan

        There are three tiers of encryption on iOS, and they stack:

        Tier 1 — iOS Data Protection (do this now, it's free). Set NSFileProtectionComplete on your records directory. The OS encrypts every file at rest using the device passcode and Secure Enclave. Files are unreadable when the device is locked. Zero code changes to your read/write logic, zero processing overhead, and it's what Apple recommends for health data.     • 4. Sync architecture

      • “Event log” + “CRDT-ish” metadata, not file overwrites

         • Every change produces an append-only event:

         ◦ upload_record, tag_added, note_created, doc_deleted, correction_made

         • Cloud stores events + encrypted blobs.

         • Devices replay events to converge state.

         • Conflicts (e.g., two devices edit the same note) resolve at metadata layer.

        This keeps you sane across iOS/Android + web.

      • User controls (these matter as much as crypto)

    • 12. Costs

      Quick cost intuition (very rough, but useful)

       • On-device heavy (local storage/index + occasional cloud LLM): cheapest cloud bill, best privacy.

       • Cloud RAG + private LLM gateway: moderate cost (you’re paying for secure infra + ops).

       • Full private model hosting (GPU always-on): most expensive, only worth it at scale or high sensitivity.

    ✅ 5. Version Control (Very Important)

    If you’re not already:

    Initialize git now.

    In your project folder:


    git init

    git add .

    git commit -m "Phase 1 working baseline"

    Then every feature:


    git add .

    git commit -m "Add PDF import"

    Now your memory lives in git history.

    This is your real “context preservation”.
